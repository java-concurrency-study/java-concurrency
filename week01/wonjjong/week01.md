# 자바 스레드의 간략한 역사
### 자바는 스레드로 만들어진다.
* 자바는 스레드 활용 기능을 내장하고 있어 특정 OS에 특화된 기능에 의존하지 않고도 동시성을 구현할 수 있었음
* 자바에서 스레드는 프로세스 안에서 실행되는 가장 작은 단위

#### 스레드
* 스레드는 하나의 프로그램 안에서 독립적인 실행 경로를 의미
* 같은 프로세스 안의 자원과 메모리를 공유 할 수 있다
  * 동일한 주소 공간을 공유하므로 프로그램의 변수와 데이터 구조에 대해 각 스레드가 동등하게 접근할 수 있음
* 프로그램 카운터, 스택, 지연 변수는 스레드 별로 할당이 되므로 각자 독립적으로 동작할 수 있다
* 결국 같은 프로세스 안에서 자원을 공유하면서 여러 작업을 동시에 처리할 수 있게 해주는 단위 
* 프로세스보다 스레드가 효율적인 이유
  * 프로세스는 각각 독립된 메로리 공간을 가지지만 스레드는 같은 프로세스 안에서 메모리와 자원을 공유할 수 있음
    - 생성 비용이 더 작다: 프로세스를 새로 만들려면 독립적으로 할당해줘야 하는 주소 공간이 더 큼
    - 데이터 공유가 용이: 프로세스는 IPC를 통해서 데이터를 주고받아야 하지만 스레드는 같은 heap을 공유하므로 데이터를 공유하기 쉽다

#### 병렬성 vs 동시성
- 병렬성
  - 여러 작업이 실제로 같은 시점에 동시에 실행되는 것 
  - 실제 동시 실행이 필요하므로 싱글 코어에서는 병렬성이 불가능함 
- 동시성
  - 여러 작업을 번갈아 가면서 수행하는 것 
  - CPU가 작업을 번갈아 처리하면서 여러 작업이 병렬로 수행되는 것 “처럼”  느껴지게 함
- 멀티코어 환경에서는 동시성과 병렬성이 함께 나타날 수 있다
  - 각 코어 안에서는 여러 작업을 번갈아 처리하는 동시성 관점
  - 여러 코어가 실제로 동시에 작업을 수행하므로 병렬성 관점도 존재

### 예외와 스레드
* 자바에서 모든 스레드는 각자의 고유한 호출 스택을 가짐
* 이 호출 스택은 스레드가 생성되어서 소멸되기 전까지의 모든 메서드의 호출 내역이 기록된다.
  
# 스레드에 숨어 있는 비용 
- 대부분의 웹 애플리케이션은 요청 하나당 스레드 하나 방식의 스레드 모델을 사용  (ex. 톰캣, 제티) 
  - 톰캣 기준으로 보면 요청이 들어왔을 때 Connector가 요청을 받고 스레드 풀의 스레드가 요청을 가져가서 응답을 반환하는 구조
  - 이 방식의 문제점은 장시간 작업을 스레드가 처리하는 경우 해당 스레드는 다른 요청을 처리하지 못한채로 묶여 있다는 것
  - 그래서 요청이 많아지고 요청에 대한 작업 소요 시간이 오래 걸리게되면 대기 시간이 늘어나면서 응답이 느려질 수 있음
  - Netty나 Webflux 같은 방식은 이벤트 루프라는 개념이 존재해서 적은 수의 스레드로 많은 비동기 요청을 처리할 수 있음
  - 가상 스레드는 기존의 스레드 모델을 사용하지만 훨씬 경량화된 스레드를 사용하여 많은 요청을 처리할 수 있도록 함 
- 최신 OS는 수백만 개의 동시 연결을 지원할 수 있음
- 리틀의 법칙에 의하면 스레드 풀에 더 많은 스레드를 생성해서 사용하면 처리량이 높아질 수 있지만 항상 높아지는 것은 아님
  - 리틀의 법칙: L = λ × W
    - L: 시스템 내 평균 작업 수
    - λ: 평균 도착률
    - W: 평균 체류 시간
    - 예를 들어 5시간 동안 300명의 손님이 왔고 평균 식사 시간이 10분인 경우 식당 내의 평균 손님 수는 아래와 같이 계산할 수 있음
      - L: 60* (1/6) = 10명
      - λ : 300 / 5 시간 = 60명 / 시간
      - W : 1/6 시간
  - 스레드도 메모리를 사용
    - 스레드도 결국 메모리에 할당되기 때문에 많아질 수록 메모리 사용량이 커짐
    - 계속해서 생성시 애플리케이션의 메모리 사용량이 늘어나 heap 메모리 초과 가능성이 존재
    - 물리 메모리 전부를 사용하게 되는 경우 디스크와의 swap이 발생하기 때문에 디스크 I/O가 발생하고, 이로인해 처리량이 낮아질 수 있음
  - 컨텍스트 스위칭 비용 존재
    - 스레드가 많아질수록 결국
 CPU도 컨텍스트 스위칭 작업을 더 자주 수행하므로 오히려 스위칭 작업이 더 커질 수 있음

#### 처리량
- 서버가 요청을 받고 처리한 후 응답을 반환하는 속도로 일반적으로 RPS 또는 TPS로 측정
- 처리량 = 처리된 전체 요청 개수 / 전체 시간 

# AtomicReference
- 스레드 사이에서 안전하게 데이터를 공유할 때 사용
- 왜 쓸까? 
  - 동시성 문제를 해결하기 위해
  - CAS(Compare And Set) 연산을 사용할 수 있음
  - 여러 스레드가 하나의 참조 값을 교체하는 상황에서 중간에 값이 바뀌지 않도록 보장해줌

# 포크/조인 풀 
- 워커 스레드가 로컬 큐에 있는 태스크를 실행하므로 캐시 선호도를 활용할 수 있음 
- 캐시 선호도: CPU 캐시의 지역성을 적극적으로 활용해서 태스크를 완료하는 개념
- Work Stealing 알고리즘
  - 포크 조인 풀에 있는 스레드들은 하나의 공유 태스크 큐를 사용하지 않고 각자의 전용 큐를 갖고 있음. 
  - 한 스레드가 자신에게 할당된 작업을 모두 마치면 다른 스레드의 끝에 있는 작업을 훔쳐와서 수행하여 CPU 사용률을 높일 수 있음 
  - 해당 알고리즘을 사용하면 오히려 다른 스레드에 있던 작업을 뺏음으로써 캐시 선호도를 활용할수 없는것이 아닐까? 
    - 캐시 선호도를 최대한 활용하되 CPU IDLE 방지를 위해서 해당 알고리즘을 사용하는 것이라고 함
    - 이를 위해 다른 스레드의 큐에 쌓인 최근 작업이 아닌 끝에 있는 작업을 가져와 수행하는 것

# 자바 동시성 제공의 변천사
1. Thread 직접 생성
   1. 직접 start() 하고 필요하면 join()을 하는 형태 
   2. 스레드 생애주기를 직접 관리해야 함
2. ExecutorService 
   1. 스레드 수 제어 및 재사용 가능, 스레드 생애주기 관리 
   2. Future.get()에서 발생하는 블로킹 문제는 여전히 존재
   3. ExecutorService는 인터페이스로 구현체로는 익숙한 ForkJoinPool, ThreadPoolExecutor
      등 많은 구현체가 존재함
3. CompletableFuture
   1. 함수형 스타일로 코드를 작성할 수 있음
   2. 내부적으로 포크/조인 프레임워크 위에서 만들어졌음 
      1. 다른 풀을 사용하도록 지정할 수도 있다
   3. 예외처리도 명시적으로 할 수 있음 
   4. 코드가 행단위로 순차 실행되는게 아니기 때문에 디버깅 작업이 더 어려울 수 있음
4. Reactive Programming 
5. 가상스레드 
   1. 플랫폼 스레드 위에서 동작
   2. 플랫폼 스레드: 자바의 전통적인 스레드, OS 스레드와 1:1로 매핑 됨
   3. 플랫폼 스레드는 포크조인풀의 지원을 받고있으므로 가상스레드도 포크조인풀의 장점을 모두 활용할 수 있음 
   4. 가상 스레드 환경에서 많은 작업을 적은 수의 워커(플랫폼 스레드)로 효율적으로 돌리는데 유리
