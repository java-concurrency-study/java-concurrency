# 한눈에 살펴보는 자바 동시성 변천사

<br>

## 1. 자바 스레드의 간략한 역사

### 자바는 스레드로 만들어진다.

- 자바의 가장 작은 실행 단위는 스레드
- 스레드는 각각의 호출 스택을 소유하고있음. 이는 스레드의 생성과 소멸되기 전까지의 모든 메서드 호출 내역이 기록됨.

### 스레드의 숨은 비용
- 서블릿 컨테이너(Tomcat, Jetty 등)가 스레드 풀에서 스레드를 꺼내 요청 전 생애주기 담당 이러한 방식은 대규모 동시 연결엔 처리량 높이는 장점이 있음.
- 그렇다고 스레드를 무한정 늘릴수잇는 것은 아님. 이유는 스레드당 힙 오부에서도 메모리를 갖게되는데 수천개의 스레드를 사용할경우 메모리를 많이 사용하게됨. 이를 고려하여야 함.
- 처리량 = 처리된 요청 수 / 전체 시간
- 또한 운영체제의 네이티브 스레드를 감싼 것이기때문에 운영체제의 네이티브 스레드 이상으로 자바 스레드를 사용할 순 없다.
- 서버마다 사용할 수있는 스레드 수가 다르기 때문에 인프라를 고려하여 작업 해야한다.(비용또한 무시 할수 없다.)

### 이그제큐터 프레임워크
- 이그제큐터 프레임워크는 동시에 실행되는 스레드의 수를 제어, 생에주기를 효율적으로 관리하여 재사용하게하는 프레임 워크
- 이그제큐터 프레임워크의 한계점 
    - Feature.get()에서 블로킹이 발생함. 결국 블로킹의 위치를 옮길뿐 블로킹을 막지 못함.
    - 캐시 일관성 유지 부하와 거짓 공유 위험. 
    - 합성력 부족. 명령형으로 합성, 선언형 스타일로 사용이 안됨.

### 스레드 풀의 발전 - 포크/조인 풀
- 이그제큐티의 한계를 보완한 것이 포크(쪼개고)/조인(합치는) 풀
- 캐시 선호도 : 태스크를 같은 CPU 코어에서 계속 실행 -> 캐시 재활용 가능 -> 메모리 접근 시간 절약
- Work-Stealing 알고리즘
  - 각 스레드가 자기만의 전용 큐 보유
  - 자기 큐가 비면 다른 스레드 큐 끝에서 태스크 훔쳐서 실행
  - 놀고 있는 스레드 없음 → CPU 사용률 극대화


### CompletableFuture
- 앞서 말한 포크/조인 풀보다 더 발전한 버전으로 자바 8에서 도입.
- 장점
  - 콜백 지옥 없이 선언형으로 비동기 흐름 작성 가능
  - 풍부한 API로 여러 비동기 연산 연결 → 가독성·유지보수성 향상
  - 내부적으로 포크/조인 풀 + Work-Stealing 활용
- 단점
  - 명령형에 익숙한 개발자에겐 발상의 전환 필요, 학습 비용 높음
  - get()은 여전히 블로킹 — 잘못 쓰면 비동기 효과 0
  - 체이닝 스타일 코드는 디버깅이 매우 어려움

### 리액티브 프로그래밍
- CompletableFuture로도 부족한 복잡한 비동기 시나리오를 위해 등장. RxJava, Akka, Spring WebFlux 등.
- 단점 (리액티브도 공짜가 아님)
  - 가파른 학습 곡선 — Observable, 배압, 연산자 등 새 개념 산더미
  - 디버깅 어려움 — 스택 트레이스가 체이닝 지점만 보여줘서 에러 출처 찾기 힘듦
  - 지나친 복잡함 — 오버엔지니어링 유혹
  - 잠재적 불일치 — 단순 요청/응답엔 굳이 리액티브 쓸 필요 없음
  - 벤더 종속 — 프레임워크 바꾸기 현실적으로 어려움

### 자바 동시성의 혁명 — 가상 스레드 (Project Loom)
- 전통 스레드에 비해 거의 비용 없이 생성 가능 → 수백만 개도 OK
- 플랫폼 스레드 위에서 동작, 포크/조인 풀의 장점도 그대로 활용
- 핵심 특징
- 특징내용기존 코드와 매끄러운 통합Executors.newVirtualThreadPerTaskExecutor() 한 줄로 교체 가능지능적인 블로킹 처리sleep·I/O 같은 블로킹 만나면 자동으로 플랫폼 스레드에 제어권 넘김 → 다른 가상 스레드 실행, 완료 시 재개자원 효율성수백만 개 생성해도 자원 고갈 없음코드 단순성기존 블로킹 스타일 그대로 써도 성능 손실 없음 — 리액티브 공부 안 해도 됨최적 활용블로킹 중 CPU 낭비 없이 플랫폼 스레드가 계속 일함
