# 한눈에 살펴보는 자바 동시성 변천사

## 1.스레드(Thread)란 무엇인가

### 1.1 정의

스레드(Thread)는 **프로세스(Process) 내에서 실행되는 작업의 단위(실행 흐름)**이다.

즉, 하나의 프로그램(프로세스) 안에서 **여러 작업을 동시에 수행할 수 있게 해주는 최소 실행 단위**라고 볼 수 있다.



### 1.2 프로세스 vs 스레드

| 구분 | 프로세스 | 스레드 |
|------|----------|--------|
| 정의 | 실행 중인 프로그램 | 프로세스 내 실행 흐름 |
| 메모리 | 독립적인 메모리 공간 | 프로세스의 메모리를 공유 |
| 생성 비용 | 높음 | 낮음 |
| 통신 | IPC 필요 | 공유 메모리 사용 |
| 장애 영향 | 독립적 | 하나 죽으면 전체 영향 가능 |

핵심
- 프로세스 = "프로그램 하나"
- 스레드 = "그 안에서 돌아가는 작업 단위"

## 2.왜 Thread 대신 Executor를 사용하는가

### 2.1 Thread 직접 생성의문제
- 생성비용 큼
  - OS 지원 사용
  - 메모리 할당
- 스레드 개수 제어 불가
  - 무한 생성 위험
- 재사용 불가
  - 한번쓰고 끝
### 2.2 Executor 장점
```java
ExecutorService executor = Executors.newFixedThreadPool(10);
```
- d요청 1000 -> thread 10 + queue (안정적)

- 작업 실행 분리
```java
 executor.submit(() -> { ... });
```

- 결과 및 예외처리
```java
 Future<?> future = executor.submit(task);
```

- Graceful shutdown
```java
executor.shutdown();
```

즉 Executor는 스레드를 직접 생성하는 것이 아니라 효율적으로 관리

## 3. 컨텍스트 전환
컨텍스트 스위칭(Context Switching)은 CPU가 현재 실행 중인 스레드를 멈추고, 다른 스레드로 실행을 전환하는 과정이다.

자바 애플리케이션 관점에서 보면, 여러 스레드가 동시에 존재하더라도 CPU는 한 시점에 제한된 수의 스레드만 실제로 실행할 수 있다. 따라서 운영체제는 스레드들을 아주 짧은 단위로 번갈아 실행시키는데, 이때 발생하는 전환 작업이 바로 컨텍스트 스위칭이다.

즉, 컨텍스트 스위칭은 "작업 그 자체"가 아니라 작업 대상을 바꾸는 행위이다.

### 3.1 왜 스레드가 많아지면 CPU 부하가 커지는가
스레드를 많이 만든다는 것은 단순히 메모리(stack)를 더 쓰는 문제로 끝나지 않는다.
스레두 수가 많아질수록 운영체제는 더 자주 다음과 같은 일을 해야한다.
- 현재 스레드 상태저장
  - 프로그램 카운터, 레지스터값, 스택포인터, 실행중이던 문맥정보
  - 위정보를 저장하지 않으면 나중에 해당 스레드를 다시 실행할때 어디서부터 이거가야하는지 알수없음
- 다음 스레드 상태복원
  - 이제 CPU가 다음 스레드를 실행하려면, 그 스레드가 이전에 어디까지 실행됬는지 복원해야한다.
  - 이전에 저장해둔 레지스터 값로드, 스택복원, 다음 실행위치복원
  - 이 과정 역시 cpu 시간을 사용
- CPU 캐시 효율 저하
  - 이부분이 실제 성능 저하에서 매우 중요
  - CPU는 빠른 실행을 위해 캐시를 사용한다. 그런데 스레드가 바뀌면, 이전 스레드가 사용하던 데이터와 다음 스레드가 사용하는 데이터가 다를 가능성이 높다.
  - 그 결과 CPU 캐시 적중률 저하, 캐시 미스증가, 메인 메모리 접근증가, 전체실행속도 저하
  - 즉, 단순히 스레드를 바꾸는데 드는 비용뿐 아니ㄹ, 바꾼뒤에도 다시 워밍업 비용이 든다
- 스케줄링 비용
  - 운영 체제는 어떤 스레드를 다음에 실행할지 결정해야한다.
  - 이 과정에서 runnalbe 상태의 스레드관리, 우선순위고료, timeslice 소진여부확인, block/wakeup 상태변경관리 등이 필요
  - 즉, 스레드가 많을수록 os 스케줄러도 더 비싸진다.
### 3.2 왜 컨텍스트 스위칭은 부하?
컨텍스트 스위칭이 많아질수록 CPU는 실제 일을 하기보다 전환 자체에 시간으 더 많이쓰게 된다.
그래서 스레드수를 무작정 늘린다고 성능이 좋아지지는 않는다. 오히려 일정 수준을 넘어서면
- 전환 비용증가
- 캐시효율저하
- 응답시간증가
- 전체처리량감소
같은 문제가 발생

### 3.3 컨텍스트 스위칭은 언제 발생하는가
#### 3.3.1 I/O 대기가 발생했을떄
```java
String result = externalApiClient.call();
```
이 호출이 blocking 방식이면, 해당 스레드는 응답을 기다리는 동안 CPU를 계속 점유할 필요가 없다.

그래서 운영체제는 그 스레드를 대기 상태로 두고, 다른 실행 가능한 스레드로 전환한다.

#### 3.3.2 sleep()을 호출했을 때
```java
Thread.sleep(1000);
```
현재 스레드는 1초동안 실행하지 않겠다고 선언한것이므로, CPU는 다른 스레드를 실행
#### 3.3.3 lock 획득에 실패했을 때
```java
synchronized (lock) {
        // critical section
}
```
이미 다른 스레드가 lock을 잡고 있다면, 현재 스레드는 대기해야 한다.
운영체제는 이 스레드를 계속 CPU에 올려둘 이유가 없으므로 다른 스레드로 전환한다.
#### 3.3.4 yield()를 호출했을 때
```java
Thread.yield();
```
현재 스레드가 CPU를 양보하겠다는 힌트를 주는 경우에도 다른 스레드로 전환될 수 있다.
#### 3.3.5 우선순위, 인터럽트, 깨어남(wakeup) 등 스케줄링 이벤트가 발생했을 때
운영체제는 특정 이벤트에 따라 어떤 스레드를 먼저 실행할지 다시 판단할 수 있다. 이 과정에서도 전환이 발생한다.

### 3.4 ExecutorService와 연결되는 이유
이 지점에서 ExecutorService가 왜 중요한지 자연스럽게 연결된다.
new Thread()를 계속 사용하면:

- 스레드 수 제어가 어렵고
- 컨텍스트 스위칭이 과도하게 발생하며
- 시스템이 불안정해질 수 있다.

반면 ExecutorService는:
- 스레드 개수 제한
- 작업 큐 관리
- 재사용 가능한 스레드 풀 운영

을 통해 컨텍스트 스위칭 비용을 통제 가능한 수준으로 유지하게 도와준다.
즉, Executor는 단순히 개발 편의 기능이 아니라,
스레드 폭증과 전환 비용을 제어하기 위한 실질적인 성능 도구이기도 하다.


## 4, 스레드 효율을 늘리기위한 수많은 노력
스레드의 실행을 막는 블로킹은 심각한 비효율을 야기한다. 스레드를 다써도 모자랄판에 단순히 기다리는곳에 스레드를 ㄴ아비하면 시스템의 확작성은 심각하게 제한된다.

스레드 효율성을 높이기 위한 수많은 노력들, 특히 I/O 응답대기 허비되는 시간을 최소화하려는 노력이 오랜 시간에 걸쳐 자바의 진화를 이끌어옴
입출력 작업 진행 시간동안 독립적으로 다른 작업을 할 수 있게 해주는 비동기메서드호출, 스레드 재사용을 통해 스레드 생성시간을 줄여주는 스레드폴링, 
논블로킹, 이벤트 기반 상호작용을 강조하는 최근의 리엑티브 프로그래밍에 이르기까지 다양한 패러다임을 적용해왔다.

## 5. AtomicReference
- 스레드 사잉에서 안전하게 객체 참조를 공유하기위해 사용
- 왜 thread-safe 하나 ?
  - CAS 연산을 사용할 수 있음
  
### 5.1 CAS 란 ?
  - 다음 3단계를 하나의 원자 연산으로 수행
    - 현재 값 확인 (compare)
    - 기대 값과 같은지 비교
    - 같으면 새 값으로 교체 (swap)
  - non blocking !
  - 또 한 빠르다
    - lock 없고
    - thread block 없고
    - context switching이 없음
  - 한계점
    - ABA 문제 (A -> B -> A)
      - 값이 같아 보이지만 실제로는 변경됨
      - 즉, 핵심은 값이아니라 상태
      - 이 값이 한번도 안바뀌었나? 
      - CAS는 값만 비교하기떄문에 변경이력을 전혀 모른다
      - 해결 : AtomicStampedReference
    - 반복 재시도 비용
      - CAS는 실패하면 재시도한다. 즉, 경쟁이 심하면 CPU 소모증가
  - 언제 사용해야하는가
    - 단일변수 업데이트
    - 상태변경
    - lock 없이 빠른 처리 필요
  - 부적합
    - 복잡한 여러변수동기화
    - 트랜잭션 성격 작업
      - 주문상태 + 결제상태 항강 같이 변경되어야하는데 Atomic은 보장못함
## 5. 포크 조인 / 풀
멀티코어 CPU 시대에서 효율적인 병렬처리 + 캐시효율 _ 스레드관리문제를 해결하기위해 등장
다른 코어가 같은 메모리를 수정하면 캐시무효화발생
죽, 캐시라이무효화와 동기화가 빈번히 발생하면 태스크를 CPU 사이에 고르게 배분하는 이그제큐터 프레임워크 사용 시 전반적인 성능에 악영향을 줄 수있다.
이떄문에 포크 조인 / 풀 등장

### 5.1 포크 조인 / 풀이 어떻게 해결했는데?

#### 5.1.1 각 스레드가 자기 큐를 가진다.
포크/조인 풀은 워커 스레드가 로컬큐에있는 태스크를 실행하기 떄문에 캐시선호도를 활요할수 있으며,
여러 태스크가 동일한 공유데이터에 자주 접근하는 상황에서 특히 효과적
예를들어, 기존방식에는 모든 스레드가 하나의 큐에서 작업을 가져갔는데 만약 동시에접근할때 아래 결과발생
- lock
- context switching
- 성능저하
- 캐시깨짐

즉 Global Queue? 여기에 접근안하고 스레드별로 자체 로컬 큐를 가짐
- 자기 큐니까 경쟁없어지고
- 이로인해 캐시효율 증가 (서로다른 메모리접근, 캐시라인 공유없음)

- Work Stealing
  - 쉽게말하면 Thread2 놀고있어 Thread1 큐에서 작업을 훔친다

자기 큐는 기본구조이고 Work Stealing은 부하분산 전략이라고 생각하면 될듯

#### 5.1.2 CompletableFuture
CompletableFuture 비동기작업을 선언적으로 연결하고 조합하기위한 API
코드의 가독성도 증가 (콜백지옥탈피)
- 포크/조인 풀 = 실행엔진
- CompleableFuture = 그 엔진위에서 비동기 흐름을 다루는 상위 API

포크/조인 풀은 "어떻게 실행할까"에 가깝고 CompletableFuture는 "비동기 작업을 어떻게 연결하고 조합할까"에 가깝다


- 단점?
  - 디버깅이 매우 어려운 상황이 올 수 있음

## 가상스레드 등장
- 가상스레드와 플랫폼 스레드
  - 플렛폼 스레드는 포크/조인 풀의 자원을 받고 있으므로, 플렛폼 스레드위에서 동작하는 가상스레드 역시 포크/조인 풀의 장점 모두 활용가능
- 지능적인 블로킹 연산처리
  - 가상스레드는 sleep이나 네트워크 I/O 같은 블로킹 연산을 만나면 실행을 멈추고 제어권을 자동으로 플랫폼 스레드에게 넘겨줌
  - 제어권을 넘겨받은 플랫폼 스레드는 다른 가상 스레드를 실행할수있으므로 사용가능한 자원을 낭비없이 효율적으로 활용
