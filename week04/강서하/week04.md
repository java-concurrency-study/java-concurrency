# 3장. 직접 만들어보고 이해하는 최신 동시성 메커니즘

> 가상 스레드 구현을 이해하려면 ForkJoinPool과 컨티뉴에이션(continuation) 두 개념이 핵심이다.

---
## 헷갈려서 정리한 개념 계층 정리
1. 스레드
- 실제로 코드를 실행하는 주체로 CPU위에서 돌아가는 일꾼
2. 스레드 풀
- 스레드를 여러개 모아놓고 재사용 하는 관리구조를 말함. 일꾼의 집합. 인력사무소
3. 태스크
- 스레드 풀에 던져주는 일감. 인력사무소에 일을 주는 업체
4. 큐
- 태스크가 스레드에 배정되기 전 대기하는 대기열
5. 포크조인풀 
- 스레드 풀의 한 종류로 일반 스레드풀과 다른점이 있음
- 스레드마다 개별 큐를 보유하고있고 
- 워커 스틀링이라는 유휴 스레드가 다른 스레드 큐에서 태스크를 훔쳐 옴

관계도</br>
태스크(Task)</br>
  ↓ 제출</br>
큐(Queue)</br>
  ↓ 꺼내서 실행</br>
스레드(Thread)  ← 스레드풀이 관리</br>
  ↑</br>
ForkJoinPool = 스레드풀의 특별한 구현체</br></br>

## 스레드 풀 (Thread Pool)

애플리케이션 시작 시 스레드를 뭉치를 만들고 그 뭉치중 하나씩 꺼내서 쓰는 개념 </br>

스레드 풀이 필요한 이유
- 생성 가능한 스레드 수를 제어하여(운영체제 스레드 수를 넘지않음) 한꺼번에 많은 자원을 만드는 것을 방지
- 스레드를 매번 새로 만들지 않아 응답 속도가 빠름
- 복잡성이나 누수를 방지하여 비즈니스 로직에 집중할 수 있게 함

## SimpleThreadPool 직접 만들기

- 빌드되는 시점에 스레드풀이 생성되는 것이 아닌 실행시점에 스레드풀이 생성이됨.
- 정확히는 스레드풀이 자동 생성 되는 것이 아니고 , 스레드풀이 활동할 수있는 환경 즉 포크조인풀이 실행되는 시점에 생성이 됨.

## 이그제큐터 프레임워크 (Executor Framework)

- 위에서 만들었던 심플스레드큐를 잘 사용하기위한 코드꾸러미

### ThreadPoolExecutor 주요 파라미터

```java
new ThreadPoolExecutor(
    corePoolSize,       // 유휴 상태에도 유지되는 최소 스레드 수
    maximumPoolSize,    // 허용되는 최대 스레드 수
    keepAliveTime,      // core 초과 유휴 스레드가 살아있는 최대 시간
    unit,
    workQueue,          // 실행 전 태스크 보관 큐
    threadFactory,
    handler             // 스레드+큐 모두 한계 도달 시 처리
);
```

풀 크기 설정 기준:
- CPU 집중 작업 → 스레드 수 ≈ CPU 코어 수 (컨텍스트 스위칭 오버헤드 방지)
- I/O 집중 작업 → CPU 코어 수보다 많아도 유리 (대기 중 다른 태스크 처리 가능)

`Runnable`은 반환값 없음 → 결과가 필요하면 `Callable` + `Future` 사용.

## ForkJoinPool

Java 7부터 추가된 특별한 스레드 풀.</br>
work-stealing 알고리즘을 사용하여 작업을 훔쳐서 실행할수 있음</br>

### 전통적 스레드 풀과의 핵심 차이

| 항목 | 전통적 ThreadPoolExecutor | ForkJoinPool |
|------|--------------------------|--------------|
| 큐 구조 | 공유 큐 (경쟁 오버헤드 발생) | 스레드마다 개별 큐 |
| 유휴 스레드 동작 | 공유 큐에서 태스크 가져옴 | 다른 스레드 큐 뒤쪽에서 훔쳐옴 (Work-Stealing) |
| 스레드 수 | 고정적 | 작업량에 따라 동적 조정 |
| 동기화 | 명시적 락/조건변수 필요 | Work-Stealing으로 동기화 거의 불필요 |
| 태스크 의존성 | 인식 못함 (서브태스크 대기 중 블로킹) | 서브태스크 대기 중 다른 태스크 실행 가능 |

### Work-Stealing 알고리즘 동작 원리

각 워커 스레드는 덱(양방향큐) 로 만들어진 자신만의 작업 큐를 갖는다.

- 워커 스레드: 자신의 큐 앞쪽(LIFO) 에서 태스크를 꺼냄 → 스택처럼 동작 → 캐시 적중률 향상
- 도둑 스레드: 다른 스레드 큐 뒤쪽(FIFO) 에서 가장 오래된 태스크를 훔쳐옴 → 경합 최소화
- 큐 관리: 명시적 락 대신 CAS(비교 후 교체, compare-and-swap) 연산으로 동시 접근 처리

### 비동기 모드 (asyncMode)
기본 ForkJoinPool은 분할 정복(LIFO) 방식이지만, 이벤트 처리처럼 순서가 중요한 독립 태스크엔 비동기 모드(FIFO) 로 생성할 수 있다.

## 컨티뉴에이션 (Continuation)

프로그램의 현재 실행 상태를 스냅샷으로 저장하고 나중에 중단 지점에서 정확히 이어서 실행할 수 있는 메커니즘. 스레드 컨텍스트 전환과 유사하지만, 메서드 수준에서도 사용 가능하다.</br>
호출 스택이 깊을수록 비용이 많이 든다.</br>
