# 2장. 쉼 없이 CPU를 활용하는 가볍고 부지런한 가상 스레드

## 1. 가상 스레드란?

- **JVM이 직접 관리**하는 경량 스레드 (운영체제가 아님)
- **캐리어 스레드(carrier thread)** 위에서 실행됨
  - 캐리어 스레드는 ForkJoinPool에서 가져온 플랫폼 스레드
- Work-Stealing 알고리즘 + 고급 스레드 풀링 메커니즘의 장점을 물려받음

---

## 2. 가상 스레드 아키텍처

- 가상스레드 <-> 캐리어 스레드 <-> OS 스레드가 N:n:n구조로 이루어져있음.
- 가상스레드 : jvm이 만든 작업단위의 경량 스레드
- 캐리어스레드 : 플랫폼 스레드
- os 스레드 : 물리적인 하드웨어 자원
- 마운트 / 언마운트 : 가상스레드가 캐리어 스레드에 올라타는것 , 내려오는 것을 뜻함.

---

## 3. 플랫폼 스레드와의 차이점

### 3-1. 관리 주체

- 플랫폼 스레드 → OS가 직접 만들고 관리 (OS 스레드와 1:1)
- 가상 스레드 → JVM이 관리, OS는 존재도 모름

### 3-2. 메모리

- 플랫폼 스레드 → 생성 시 스택 메모리를 OS가 미리 크게 고정 할당 (보통 1MB)
- 가상 스레드 → 힙에 저장, 수백 바이트부터 시작해서 필요한 만큼만 동적으로 늘어남

→ 플랫폼 스레드 1,000개 = 약 1GB / 가상 스레드 1,000개 = 수 MB 수준

### 3-3. 생성 비용

- 플랫폼 스레드 → OS한테 스레드 만들어달라고 시스템 콜 해야 함 → 느리고 비쌈
- 가상 스레드 → JVM이 힙에 객체 만드는 수준 → 매우 싸고 빠름

### 3-4. 블로킹 시 동작

- 플랫폼 스레드 → 블로킹 만나면 OS 스레드째로 멈춤 → 그 자원을 그대로 점유
- 가상 스레드 → 언마운트해서 캐리어 스레드 반납 → 캐리어는 다른 일 하러 감

### 3-5. 스케줄링 오버헤드

- 플랫폼 스레드 → OS가 스케줄링 → 컨텍스트 스위칭 발생 (레지스터, 스택 포인터 등 저장/복원, 비쌈)
- 가상 스레드 → JVM이 스케줄링 → OS 개입 없이 힙의 스택 프레임만 바꿔치기 → 훨씬 가벼움

---

### 4. 가상 스레드의 4가지 강점

1. **가벼움** — 수백만 개 생성해도 시스템 자원 고갈 없음
2. **스케줄링** — JVM이 직접 스케줄링 → CPU 낭비 최소화
3. **블로킹 허용 능력** — 블로킹 연산 시 캐리어 스레드를 반납, 다른 가상 스레드가 실행 계속
4. **매끄러운 통합** — 기존 코드 패턴/추상화 그대로 사용 가능

---
## 5. 가상 스레드 생성 방법
```
javac 05_BasicVirtualThread.java && java 05_BasicVirtualThread
```


### 5-1) 가상 스레드만의 고정된 특성

- **스레드 그룹**: 모든 가상 스레드는 단 하나의 그룹(`VirtualThreads`)에 속함
  - `getThreadGroup()` 호출 시 항상 동일한 인스턴스 반환
- **우선순위**: 항상 `NORM_PRIORITY` — `setPriority()` 호출해도 변경 안 됨
- **데몬 상태**: 항상 데몬 스레드 — `setDaemon()` 호출해도 효과 없음


### 5-2) 제약 사항

- `Thread::getAllStackTraces` — 플랫폼 스레드 스택만 반환, 가상 스레드 포함 안 함
- 어떤 플랫폼 스레드가 가상 스레드를 실행 중인지 확인하는 방법 없음

## 6. 가상 스레드 생성 시연

10,000개 작업을 동시에 실행해도 가상 스레드는 아무 문제 없음:

```
javac Ex06_VirtualThreadCreationDemo.java && java Ex06_VirtualThreadCreationDemo
```

알수있는 결과값 - 플랫폼 스레드는 OS 자원이 한계, 가상 스레드는 힙 메모리가 한계 — 힙이 훨씬 넉넉하니까 가상 스레드가 압도적으로 많이 만들 수 있음

## 7. 처리량과 확장성

- 가상 스레드는 **빠르게** 실행되도록 설계된 게 아님
- **더 뛰어난 확장성(scalability)** 을 제공하도록 설계됨
- 핵심 원리: **리틀의 법칙(Little's Law)**

### 7-1) 리틀의 법칙

$$\lambda = \frac{N}{d}$$

| 변수 | 의미 |
|------|------|
| λ (람다) | 처리량 — 단위 시간당 완료되는 항목 수 |
| N | 동시성 — 동시에 처리 중인 항목의 평균 수 |
| d | 응답 시간 — 단일 항목 처리에 걸리는 평균 시간 |

### 7-2) 가상 스레드와 리틀의 법칙

- 전통적인 플랫폼 스레드: N을 늘리는 데 OS 수준의 제약이 있음
- 가상 스레드: **프로그래밍 모델 변경 없이** 훨씬 더 높은 N 확보 가능
- **d를 줄이지 않고도 λ를 높일 수 있다** → 가상 스레드의 핵심 가치

### 7-3) 가상 스레드가 특히 효과적인 상황

- ✅ **높은 동시 작업 수** (수천 개 이상의 동시 요청)
- ✅ **CPU 집중적이지 않은 작업** (I/O 대기, DB 쿼리, 네트워크 호출)
- ❌ **CPU 집중적인 작업** — 가상 스레드로 해결 불가

## 8. 가상 스레드 내부 동작 방식

### 8-1) 스택 프레임과 메모리 관리

| | 플랫폼 스레드 | 가상 스레드 |
|--|------------|-----------|
| 스택 저장 위치 | OS가 할당하는 **일체형 메모리 블록** | **힙(heap)** — 가비지 컬렉션 대상 |
| 스택 크기 예측 | 미리 예측 필요 | 불필요 |
| 메모리 | 수십 MB | 수백 바이트에서 시작, 동적 조정 |

### 8-2) 캐리어 스레드와 운영체제의 개입

- 운영체제는 **가상 스레드의 존재를 모름**
- JVM이 가상 스레드를 캐리어 스레드(플랫폼 스레드)에 **마운트(mount)** 해서 실행
- 캐리어 스레드는 **특화된 ForkJoinPool** 소속

### 8-3) 블로킹 연산 처리

1. 가상 스레드가 I/O 대기 등 블로킹 연산 만남
2. 캐리어 스레드에서 **언마운트(unmount)**
3. 힙의 스택 프레임 내용 보존
4. 캐리어 스레드는 **자유로운 몸**이 되어 다른 가상 스레드 처리
5. 블로킹 완료 후 다시 마운트 → 실행 재개

→ JDK 내의 거의 모든 블로킹 지점에 소급 적용됨

### 8-4) 투명성과 비가시성

- 마운트/언마운트 과정은 **자바 코드에서 보이지 않음**
- 캐리어 스레드의 `ThreadLocal` 값도 가상 스레드에게는 보이지 않음
- 기존 코드베이스를 변경할 필요 없이 가상 스레드 사용 가능

---

## 9. 비동기 연산 단순화

가상 스레드를 쓰면 비동기 연산과 태스크 집계(aggregation)를 훨씬 쉽게 할 수 있다.  
가상스레드가 .get()을 호출한다는 것은 캐리어스레드를 반납한다는 이야기, 그 캐리어스레드는 다른 가상스레드를 실행시키겠지? 즉 효율적으로 처리한다는 이야기.

### 9-1) 동시 처리 (invokeAll 활용)

결과 타입이 동일한 여러 태스크를 한 번에 처리할 때 `invokeAll()` 사용:
nodejs에서의 promiseAll과 비슷한 의미. 다만 병렬 처리를 하는 주체가 이벤트루프이냐, 가상스레드이냐가 차이.

## 10. 든든한 구조적 동시성

`Future::get`은 예외를 던질 수 있고, 타임아웃 처리가 없으면 무한 블로킹 위험 → **구조적 동시성**으로 해결.

=> get()을 하게되면 캐리어스레드를 반납하니 그 캐리어 스레드는 다른 가상스레드에게 자원을 . 수있다. 즉 놀지않고 일을 한다. 이것을 구조적 동시성으로 실행한다는 뜻인가? => 마운트, 언마운트에 대한 이야기였음.

=> 여러 작업의 생명주기를 안전하게 묶어 관리한다는 이야기?
 

팀장이 팀원들한테 일 시키는 상황

팀장(메인 가상 스레드)
├── 팀원 A → API 호출
├── 팀원 B → DB 조회
└── 팀원 C → 파일 읽기

구조적 동시성 없을 때 (Future):
팀원 A가 중간에 쓰러짐 (예외 발생)
→ 팀장은 get()에서 예외 받음
→ 근데 팀원 B, C는 아무도 알려주지 않아서 계속 일하고 있음
→ 자원 낭비, 결과도 필요 없는데

구조적 동시성 있을 때:
팀원 A가 중간에 쓰러짐
→ 팀장이 자동으로 팀원 B, C한테 "그만해" 신호
→ 다 같이 정리하고 종료
---

### 10-1) 구조적 동시성의 이점

- **fail-fast**: 한쪽 subtask 실패 시 해당 영역 안의 모든 태스크 즉시 취소
- 예외 전파와 타임아웃 처리를 언어 구조 수준에서 캡슐화
- 개발자는 스레드 생성/관리/취소 로직 대신 고수준 작업에만 집중 가능

## 11. 요청 제한을 통한 자원 제약 관리

가상 스레드는 사실상 무제한의 동시성 → 오히려 백엔드 자원(DB, 외부 API 등)이 과부하될 위험.  
플랫폼 스레드 시절에는 스레드 풀 크기가 자동으로 요청 제한 역할을 했지만, 가상 스레드는 그게 없음.  
→ **세마포어(Semaphore)** 로 명시적인 요청 제한 필요.

### 11-1) 자바의 세마포어 이해

세마포어는 **출입증(permit)** 을 관리하는 동기화 메커니즘 (`java.util.concurrent.Semaphore`).

| 메서드 | 설명 |
|--------|------|
| `acquire()` | 출입증 요청. 없으면 대기 |
| `release()` | 출입증 반납. 대기 중인 스레드가 받아 진행 |
| `availablePermits()` | 현재 사용 가능한 출입증 개수 반환 |

### ResourcePool: 세마포어로 DB 연결 풀 관리

```java
public class ResourcePool {
    private final Semaphore semaphore; // ❶ 동기화 담당

    public ResourcePool(int resourceCount) {
        this.semaphore = new Semaphore(resourceCount); // ❷ 커넥션 풀 크기 = 출입증 수
    }

    public Optional<String> useResource(String query) {
        try {
            semaphore.acquire(); // ❸ 요청 제한 지점
            try {
                return queryDatabase(query); // ❹ 임계 영역
            } finally {
                semaphore.release(); // ❺ 반드시 finally에서 반납
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt(); // ❻ 인터럽트 상태 보존
            return Optional.empty();
        }
    }
}
```

### 11-2) 세마포어 사용 시 유의사항

| 항목 | 내용 |
|------|------|
| **공정성** | `new Semaphore(n, true)` → FIFO 보장, 단 성능 약간 저하 |
| **블로킹** | 가상 스레드 도입 후 블로킹 오버헤드 크게 감소 |
| **오류 처리** | `finally` 블록에서 반드시 `release()` 호출 |
| **출입증 관리** | 출입증을 획득하지 못한 스레드도 `release()` 호출 가능 → 의도치 않게 출입증 개수 증가 위험 |
