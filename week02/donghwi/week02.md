# 2. 쉼 없이 CPU를 활용하는 가볍고 부지런한 가상 스레드

## 2.1 가상스레드
가상스레드는 캐리어 스레드위에서 실행되는데, 캐리어 스레드는 본질적으로 포크/조인 풀에서 가져온 스레드다.
이런 설꼐덕분에 가상스레드는 고급 스레드 폴링 매터니즘과 효율적인 WorkStealing 알고리즘의 장점을 물려받음

- 캐리어스레드
- 가상스레드 스케줄러의 병렬성은 플렛폼 스레드의 수를 의미, 기본값은 시스템에서 사용 가능한 프로세서의 수로 설정
- 플렛폼 스레드
  - 네이티브 스레드 (운영체제 스레드
  - 자바스레드와 커널스레드 사이의 일대이 관계를 유지
- 가상스레드
  - 플렛폼 스레드와는 다르게 JVM에서 관리, 수백만 개의 스레드를 생성하더라도 시스템 자원이 고갈되지않는다.
  - 가상 스레드는 직접적으로 커널스레드와 매핑되지 않고 다수의 가상스레드가 적은 수의 캐리어 스레드 풀을 공유
  - 플렛폼 스레드에 비해 훨씩 적은양의 메모리를 사용 (시스템 자우너 더 적게 소모)
  - JVM에 의해 스케줄링되므로 CPU 사이클을 낭비없이 더 잘 사용
  - 블로킹 허용. sleep 메서드 호출 같은 블로킹 연산을 수행할 떄도 성능저하 발생안함. 시스템 자원을 점유하지않고 제어권을 캐리어스레드에게 넘겨서 다른 가상스레드들이 계속 효율적으로 실행
  

## 2.2 가상스레드는 왜 다 데몬스레드 ?
```java
일반 스레드 → JVM 종료 막음
데몬 스레드 → JVM 종료 막지 않음
```
가상스레드는 작업단위라서 JVM lifecycle에 영향을 주면 안됨
만약 가상스레드가 non-daemon이면 ?
```java
가상 스레드 10만개 생성
-> 하나라도 안끝나면 JVM 종료안됨
-> 시스템 망함
```
중요한 포인트! -> 가상스레드는 절대 JVM 생명주기에 영향을 주지 않는다.

## 2.3 가상스레드에 사용에 적합한것과 부적합한것
가상스레드가 동시성 문제의 만병통치약은 아니라는점에 주목
- 적합한것
  - 높든 동시작업수 (네트워크 IO, DB/Redis, 파일IO)
- 부적합것
  - CPU 집중적인 작업 수행
  - 예를들어 영상처리, 복잡한계산, 압축/암호화 헤비한 작업, 대량 json 파싱, 이미지 리사이징
가상스레드는 디스크를 빠르게 만드는 기술이아니라 디스크 기다리는 동안 다른 일을 하게 만드는 기술

가상 스레드는 "기다리는 작업"을 위한 기술 CPU 작업은 결국 코어 개수만큼 처리됨.
CPU 작업은 플랫폼 스레드, I/O는 가상스레드 이 말은 개념적으로는 맞는것같지만, 실제로는 명시적으로
프랫폼 스레드를 직접 다루기보다 fixed thread pool로 격리하는 식으로 구현하는 경우가 많음

### I/O와 CPU 작업이 섞여 있을 때 권장 패턴

실무에서는 I/O와 CPU 작업이 하나의 흐름 안에 같이 존재하는 경우가 많다.
예를 들어 LLM API를 호출(I/O)하고 → 결과를 분석(CPU)하고 → DB에 저장(I/O)하는 흐름이 그렇다.

이럴 때 **두 개의 Executor를 분리**하는 것이 핵심이다.

```
가상 스레드 Executor (I/O 담당)         플랫폼 스레드 풀 (CPU 담당)
──────────────────────────────         ──────────────────────────
수백만 개 생성 가능                      CPU 코어 수만큼만 생성
블로킹 중 캐리어 스레드 반납              CPU를 계속 점유하는 게 맞음
callLlmApi(), saveResult() 적합         heavyScoreCalculation() 적합
```

```java
ExecutorService ioExecutor = Executors.newVirtualThreadPerTaskExecutor();

ExecutorService cpuExecutor =
    Executors.newFixedThreadPool(Runtime.getRuntime().availableProcessors());

ioExecutor.submit(() -> {
    // 1. I/O - 가상 스레드가 API 응답 기다리는 동안 캐리어 스레드 반납
    String llmResult = callLlmApi();

    // 2. CPU - CPU 집중 작업은 플랫폼 스레드 풀에 위임
    Future<ScoreResult> future = cpuExecutor.submit(() -> heavyScoreCalculation(llmResult));
    ScoreResult score = future.get(); // 가상 스레드가 여기서 블로킹 → 캐리어 반납

    // 3. I/O - 다시 가상 스레드가 처리
    saveResult(score);
});
```

**왜 CPU 작업을 가상 스레드에서 직접 하면 안 되나?**

가상 스레드가 CPU 작업을 수행하는 동안은 블로킹이 없으므로 캐리어 스레드를 계속 점유(pinning)한다.
가상 스레드를 1000개 띄워도 결국 캐리어 스레드(= CPU 코어 수)만큼만 동시 실행되는데,
여기에 스케줄링 오버헤드까지 추가되어 오히려 플랫폼 스레드보다 느려질 수 있다.

> 타임아웃을 추가하면 더 견고하다
> ```java
> ScoreResult score = future.get(30, TimeUnit.SECONDS);
> ```

## 2.4 리틀의법칙
| 기호 | 의미 |
|------|------|
| L | 동시에 처리 중인 작업 수 (동시성, concurrency) |
| λ (lambda) | 초당 요청 수 (throughput, RPS) |
| W | 한 작업의 처리 시간 (latency) |
### 🍔 직관적인 비유 (햄버거 가게)
처리량 : 10 * 5 = 50                                                                                                                            
- 초당 주문: 10명 (λ = 10)
- 한 명 처리 시간: 5초 (W = 5)

리틀의 법칙은 "시스템이 동시에 몇개르 ㄹ처리해야 하는지" 알려주는 공식
가상스레드는 그 동시성을 현실적으로 가능하게 만들어주는 기술
즉 d(응답시간)을 줄이지 안혹도 L을 틀릴 수 있는 비결

## 2.5 실제 운영환경에서의 성능 분석

리틀의 법칙을 기반으로 한 기초 코드는 가상 스레드의 효과를 보여주지만, **실제 운영환경**에서 정확한 성능 분석을 하려면 아래 사항들을 고려해야 한다.

### 고려해야 할 개선사항

| 항목 | 설명 |
|------|------|
| JVM 워밍업 | JIT 컴파일러가 최적화를 완료하기 전의 초기 실행 결과는 신뢰하기 어렵다. 충분한 워밍업 후 측정해야 한다. 자세한 내용은 아래 참고. |
| 반복 벤치마크 | 1회 실행 결과는 노이즈가 많다. 여러 번 반복 후 평균/표준편차 등 통계 분석이 필요하다. |
| 메모리 모니터링 | 처리량만 보면 안 된다. 메모리 사용량을 함께 보아야 자원 트레이드오프를 파악할 수 있다. |
| 현실적인 I/O 시뮬레이션 | 일정한 지연이 아닌 **지연 시간 변동(jitter)** 을 적용해야 실제 네트워크/DB 환경과 유사하다. |
| 확장성 테스트 | 작업량을 점진적으로 늘려가며 시스템이 어느 지점에서 한계에 도달하는지 파악해야 한다. |
| 타임아웃 및 오류 처리 | 일부 작업이 실패하거나 지연될 때의 동작도 측정에 포함해야 견고한 분석이 된다. |

### JVM 워밍업이란?

JVM은 처음 코드를 실행할 때 **인터프리터 모드**로 동작한다.
같은 코드가 반복 실행되면 JIT(Just-In-Time) 컴파일러가 해당 코드를 **네이티브 코드로 컴파일**해서 훨씬 빠르게 만들어준다.

```
1회 실행  → 인터프리터 모드 (느림)
2회 실행  → 인터프리터 모드 (느림)
...
N회 실행  → JIT 컴파일 발동 → 네이티브 코드 변환 → 빠름
```

즉, **처음 몇 번의 실행 결과는 JIT 최적화가 완료되기 전**이라 실제 성능을 반영하지 못한다.

#### 벤치마크에서 워밍업을 적용하는 이유

```
워밍업 1~3회  →  JIT이 최적화하도록 버려지는 실행 (측정 제외)
벤치마크 1~5회 →  JIT 최적화 완료 후 진짜 측정
```

> Ex04에서 `WARMUP_ROUNDS = 3`으로 먼저 실행하고, 이후 `BENCHMARK_ROUNDS = 5`만 통계에 반영하는 이유가 바로 이 때문이다.

#### 비유로 이해하기

요리사가 처음 주문을 받으면 레시피를 보면서 천천히 만든다.
하지만 같은 메뉴를 10번 만들면 레시피 없이도 빠르게 만들 수 있다.
JVM도 마찬가지로, 처음엔 코드를 해석하면서 실행하다가 반복되면 외워서(컴파일해서) 빠르게 실행한다.

### JMH (Java Microbenchmark Harness)

중요한 성능 평가에는 `JMH`와 같이 널리 사용되는 마이크로벤치마킹 프레임워크 사용을 권장한다.

- JVM 워밍업 자동 처리
- 통계적으로 신뢰할 수 있는 반복 측정
- 다양한 벤치마크 모드 지원 (처리량, 평균 시간, 샘플링 등)
- JIT 최적화의 영향을 최소화하는 설계

> 단순 `System.currentTimeMillis()` 측정은 JIT, GC, OS 스케줄링 등의 영향을 받아 부정확하다.
> 실제 성능 분석이 필요한 경우 반드시 JMH를 사용하자.
