# 가볍고 부지런한 가상 스레드

<br>

## 6. 가상 스레드의 한계

- 가상 스레드에는 `Pinning`이라고 알려진 제약사항이 있음

#### 💡 Pinning

- `Pinning`은 가상 스레드가 자신의 캐리어 스레드에 묶여서 고정되는 상황을 의미함
- `Pinning`된 상태에 빠진 가상 스레드는 **Blocking I/O**를 만나면 캐리어 스레드로부터 **unmount**할 수 없으며, 고정되어 있는 동안 캐리어 스레드를 점유하게 됨

<br>

### 6-1. synchronized

- 가상 스레드가 synchronized 키워드가 선언되어 있는 블럭이나 메서드에 진입할 때 `Pinning`됨

#### 원인
- synchronized는 `모니터`라는 개념을 사용하는데, **JVM**은 현재 어떤 스레드가 `모니터`를 보유하고 있는지 추적함, 여기서 추적하는 스레드는 가상 스레드가 아니라 캐리어 스레드(= 플랫폼 스레드)임, 그렇기 때문에 가상 스레드가 객체의 `모니터`를 획득하면 **JVM**은 가상 스레드가 아닌 가상 스레드가 실행되고 있는 플랫폼 스레드가 `모니터`를 획득한 것으로 기록함

#### JDK 24 이후

- JDK 24 이후부터는 synchronized 내부에서 발생하는 `Pinning`은 없어짐
- [The reason for pinning](https://openjdk.org/jeps/491)
- JDK 24 이전을 사용한다면 `ReentrantLock`을 사용하여 고정 문제를 피할 수 있음

#### ReentrantLock

- ReentrantLock는 내부적으로 `AQS(AbstractQueuedSynchronizer)` 기반으로 구현되어 있기 때문에 LockSupport.park() 또는 LockSupport.unpark() 메서드를 사용함
- ReentrantLock는 `AQS(AbstractQueuedSynchronizer)`로 인해 **JVM**은 가상 스레드가 언제 파킹되는지 감지할 수 있으며, 파킹을 감지하면 캐리어 스레드로부터 가상 스레드를 **unmount**할 수 있음

<details>
	<summary>예제 코드</summary>

```java
synchronized 키워드를 사용한 예제
public class ThreadPinnedExample {

    private static final Object lock = new Object();

    public static void main(String[] args) {

        List<Thread> threads = IntStream.range(0, 10)
                .mapToObj(it -> Thread.ofVirtual().unstarted(() -> {
                    if (it == 0) {
                        System.out.println(Thread.currentThread());
                    }

                    synchronized (lock) {
                        try {
                            Thread.sleep(25);
                        } catch (InterruptedException e) {
                            Thread.currentThread().interrupt();
                        }
                    }

                    if (it == 0) {
                        System.out.println(Thread.currentThread());
                    }
                }))
                .toList();

        threads.forEach(Thread::start);
        threads.forEach(it -> {
            try {
                it.join();
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        });
    }
}

// 결과
VirtualThread[#23]/runnable@ForkJoinPool-1-worker-1
VirtualThread[#23]/runnable@ForkJoinPool-1-worker-1

ReentrantLock를 사용한 예제
public class ThreadPinnedExample {

    private static final ReentrantLock lock = new ReentrantLock();

    public static void main(String[] args) {

        List<Thread> threads = IntStream.range(0, 10)
                .mapToObj(it -> Thread.ofVirtual().unstarted(() -> {
                    if (it == 0) {
                        System.out.println(Thread.currentThread());
                    }

                    try {
                        lock.lock();
                        Thread.sleep(25);
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    } finally {
                        lock.unlock();
                    }

                    if (it == 0) {
                        System.out.println(Thread.currentThread());
                    }
                }))
                .toList();

        threads.forEach(Thread::start);
        threads.forEach(it -> {
            try {
                it.join();
            } catch (InterruptedException e) {
                throw new RuntimeException(e);
            }
        });
    }
}

// 결과
VirtualThread[#23]/runnable@ForkJoinPool-1-worker-1
VirtualThread[#23]/runnable@ForkJoinPool-1-worker-8
```
</details>

### 6-2. 네이티브 메서드(JNI) 또는 외부 함수(Foreign Function)

- 가상 스레드가 **JNI**나 외부 함수를 만나면 `Pinning`됨

#### 원인

- **JVM**이 네이티브 코드 실행을 검사하거나 제어할 수 없기 때문에 가상 스레드가 네이티브 메서드를 호출하면 고정됨
- 네이티브 코드는 스레드 간에 마이그레이션될 수 없는 스레드 로컬 상태를 가질 수 있으며, 네이티브 호출 스택은 자바 스택 프레임처럼 저장하고 복원할 수 없고, 네이티브 코드는 OS 수준의 스레드 기본 요소와 직접 상호작용할 수 있음
- 이러한 제약사항으로 인해 가상 스레드는 네이티브 호출 전체 구간 동안 자신의 캐리어 스레드에 **mount**된 상태를 유지하게 됨

<br>

## 7. 가상 스레드에서의 여러 문제

### 7-1. ThreadLocal 변수의 문제

- ThreadLocal은 변수를 저장한 현재 스레드만 해당 값을 조회하거나 변경할 수 있음
- 가상 스레드는 매우 많이 생성될 수 있으므로, ThreadLocal에 값을 저장하면 각 가상 스레드마다 별도의 값이 저장될 수 있기 때문에 그만큼 메모리 사용량이 증가함
- 가상 스레드는 많이 생성이 되고, 생성된 개수 만큼 ThreadLocal이 만들어지니까 그만큼의 메모리가 더 사용됨
- `InheritableThreadLocal`을 사용하면 부모 스레드의 값이 자식 스레드로 전달되므로, 가상 스레드가 많이 생성되면 추가적인 메모리 사용됨

#### 플랫폼 스레드

- 보통 N개의 플랫폼 스레드를 재사용하면서 여러 요청을 처리함
- 같은 스레드 풀의 플랫폼 스레드에서는 `ThreadLocal`을 재사용함 (ex: spring security)
    - 특정 플랫폼 스레드에서 remove() 메서드를 호출하지 않으면 Thread-Safe하지 않는 문제 발생하곤함

```txt
요청1 → 요청2 → 요청3
```

<br>

#### 가상 스레드

- 매 요청마다 새로운 가상 스레드를 생성하고, 특정 작업이 끝나면 종료되는 방식으로 사용한다면 `LocalThread`는 매번 새로 만들어지기 때문에 메모리가 추가적으로 사용됨

```txt
요청1 → virtual thread A (새로 생성)
요청2 → virtual thread B (새로 생성)
요청3 → virtual thread C (새로 생성)
```

<br>

### 7-2. MDC

- `MDC`도 **ThreadLocal** 기반이기 때문에 각 가상 스레드마다 별도의 로그 정보가 쌓일 수 있음
- A 가상 스레드에서 B 가상 스레드로 실행 흐름이 넘어간다면 MDC값이 전파되지 않아 `traceId`와 같은 로그 정보들이 유실될 수 있음

<br>

### 7-3. Transaction

- `Transaction`도 **ThreadLocal** 기반이기 때문에 다른 스레드로 넘어간다면 트랜잭션 컨택스트가 사라짐

#### 🧨 문제가 발생하는 케이스

```java
케이스 1
@Transactional
public void process() {
    // 트랜잭션 있음

    Thread.startVirtualThread(() -> {
        repository.save(...); // ❌ 트랜잭션 없음
    });
}

케이스 2
@Transactional
public void process() {
    // 트랜잭션 있음

    CompletableFuture.runAsync(() -> {
        repository.save(...); // ❌ 트랜잭션 없음
    });
}

케이스 3
@Transactional
public void process() {
    // 영속성 있음
    Entity e = repository.findById(id);

    Thread.startVirtualThread(() -> {
        e.update(); // ❌ 영속성 컨텍스트 없음
    });
}
```

<br>

#### 💡 안전한 케이스

- 가상 스레드에서 시작된 트랜잭션은 **Blocking I/O**로 인해 **unmount**되었다가 이후 다른 캐리어 스레드에 **remount**되더라도, 동일한 가상 스레드의 `ThreadLocalMap`에 트랜잭션 정보가 유지되므로 계속 이어서 사용할 수 있음
- 그 이유는 트랜잭션 컨텍스트가 캐리어 스레드가 아니라, 가상 스레드의 `ThreadLocalMap`에 저장되기 때문임
- 따라서 캐리어 스레드가 바뀌더라도, 같은 가상 스레드 내부의 실행이라면 기존 트랜잭션 컨텍스트를 계속 사용할 수 있음
    - (🧨 새로운 가상 스레드로 작업이 분리되면 기존 트랜잭션 컨텍스트는 전파되지 않음)

```java
public void process() {

    Thread.startVirtualThread(() -> {
        service.process(); // 여기서 @Transactional 시작
    });
}

Thread (virtual thread)
 └── ThreadLocalMap
       ├── Connection
       ├── Transaction 상태
       └── 기타 context
```

## 8. 모니터링

### 8-1. LocalThread 모니터링

- `-Djdk traceVirtualThreadLocals` 옵션을 통해 가상 스레드가 소스 코드의 어디에서 **LocalThread**를 사용하고 있는지 추적 가능

```txt
java -Djdk traceVirtualThreadLocals ThreadLocalExample.java
```

<br>

### 8-2. Pinned 현상 모니터링

- `-Djdk tracePinnedThreads=full` 옵션을 통해 **모니터**를 보유하고 있는지 파악 가능
- full 설정은 전체 스택 트레이스 출력, short는 문제가 되는 특정 범위만 출력

```txt
java -Djdk tracePinnedThreads=full ThreadPinnedExample.java
```
