# 1. 플랫폼 스레드와 가상 스레드

## 1-1. 플랫폼 스레드

플렛폼 스레드 = 네이티브 스레드 = 운영체제 스레드

<특징>
- 운영체제에 의해 실행되는 무거운 스레드
- 스케줄링과 관리를 운영체제에 의존
- 자바스레드와 커널 스레드 사이의 일대일 관계를 유지

<추가항목>
플랫폼 스레드와 캐리어 스레드는 다른가?
-> 같은것이다. os 스레드를 감싼 자바 스레드를 일반적으로 부르는 것을 플랫폼 스레드,
플랫폼 스레드에서 가상 스레드를 마운트 한 스레드가 캐리어 스레드다.


## 1-2. 가상 스레드

가상 스레드 = 사용자 모드 스레드 = 경량 스레드

<특징>
- jvm에 의해 관리
- 캐리어 스레드 위에서 실행됨
- 직접적으로 커널 스레드와 매핑되지 않고 캐리어 스레드 풀을 공유하여 실행됨 -> 다중화

## 1-3. 플랫폼 스레드 vs 가상 스레드 비교

| 항목 | 플랫폼 스레드 | 가상 스레드 |
|-----|------------|------------|
| 가벼움 | 스레드당 MB 단위 메모리 소모, 수천 개 수준이 한계 | KB 단위로 훨씬 가볍고, 수백만 개 생성해도 자원 고갈 없음 |
| 스케줄링 | 운영체제가 담당, OS 스케줄링 오버헤드 발생 | JVM(ForkJoinPool)이 담당, CPU 사이클 낭비 없이 효율적 |
| 블로킹 허용 | blocking 시 OS 스레드를 점유한 채 대기 → 성능 저하 | blocking 시 캐리어 스레드를 반환하고 대기 → 성능 저하 없음 |

## 1-4. 캐리어 스레드가 ForkJoinPool 기반인 이유와 Work-Stealing 이점

### ForkJoinPool이란?

가상 스레드의 스케줄러는 FIFO 모드의 work-stealing ForkJoinPool 이다.
일반적인 ThreadPoolExecutor와는 구조가 다르다.
(중앙 공유 큐 1개에 작업이 쌓이고, 여러 스레드가 꺼내가는 방식. 때문에 경합이 생기거나, 유휴 스레드가 생기거나 하는 문제가 발생하지만 구조가 단순하다는 이점이 있음)

| 항목 | ThreadPoolExecutor | ForkJoinPool |
|------|-------------------|--------------|
| 작업 큐 | 중앙 공유 큐 1개 | 워커별 개별 deque |
| 로드 밸런싱 | 수동 관리 | Work-stealing 자동 처리 |
| 유휴 스레드 처리 | 대기 | 다른 워커에서 작업 훔침(Work-Stealing) |

### Work-Stealing 알고리즘이란?

각 캐리어 스레드는 자신만의 deque(양방향 큐)를 가진다.

```
// VT-1, VT-2, VT-3는 가상 스레드를 의미
캐리어 스레드 1: deque [VT-1, VT-2, VT-3]  ← 자신은 head(앞)에서 pop
캐리어 스레드 2: deque []  (유휴)           → 다른 스레드 tail(뒤)에서 steal
```

- 소유자 스레드: 자신의 deque head에서 pop (LIFO)
- 유휴 스레드: 다른 스레드의 deque tail에서 poll (FIFO, "훔침")
- 반대 방향으로 접근하여 경합을 최소화한다.

### Mount/Unmount 사이클과 Work-Stealing의 연계

```
T0: 캐리어 스레드 C1이 가상 스레드 VT-1을 마운트하여 실행 중
T1: VT-1이 blocking I/O(소켓 read 등)를 만남
    → VT-1 Unmount (스택을 힙으로 이동, C1 즉시 해제)
T2: C1이 work-stealing으로 큐에서 VT-2를 꺼내 바로 실행
T3: VT-1의 I/O 완료 → VT-1을 스케줄러 큐에 재제출
T4: C1 또는 다른 캐리어가 VT-1을 다시 마운트하여 실행 재개
```

→ 캐리어 스레드는 유휴 상태 없이 항상 실행 중인 가상 스레드를 처리 가능

### 그래서 가상 스레드가 ForkJoinPool 기반 캐리어 위에서 얻는 이점은?

1. 자동 로드 밸런싱
- 유휴 캐리어가 생기면 work-stealing으로 즉시 다른 작업을 가져옴
- 수동으로 스레드 풀 크기를 튜닝할 필요 없음

2. CPU 활용률 극대화
- 가상 스레드가 I/O로 blocking 되는 순간 캐리어 스레드를 반환
- 반환된 캐리어는 즉시 다른 가상 스레드 실행
- 캐리어 수 = CPU 코어 수만으로도 수백만 개의 가상 스레드를 처리 가능

3. 동적 병렬성 확장 (Parallelism Compensation)
- blocking I/O가 많이 발생하면 JVM이 자동으로 캐리어 스레드를 임시 추가
- I/O 작업 완료 후 원래 수로 복구
- 책에 나온 것처럼 시스템 프로퍼티 jdk.virtualThreadScheduler.parallelism으로 기본값을 조정 가능하다는 점도 특징

4. 캐시 친화성
- FIFO 모드에서 소유자(head)와 다른 스레드(tail)이 반대 방향 접근하여 경합이 감소
- 가상 스레드 간 마이그레이션 최소화 → CPU 캐시 미스를 감소시킬 수 있음

### Pinning 주의사항

캐리어 스레드에 가상 스레드가 고정(pin) 되면 unmount가 불가능해진다.

- 예시 1: synchronized 블록 내에서 blocking 할 때
```java
synchronized (lock) {
    // 이 1초 동안 캐리어 스레드가 점유
      Thread.sleep(1000);
  }
  ```
- 예시 2: native 메서드 호출 중일 때
이건 jvm 밖의 영역이라 그럼. 물론 이걸 사용하는 케이스는 보진 못했다. 

왜 unmount가 안되냐면, synchronized가 모니터 락을 캐리어 스레드 레벨에서 잡고 있어서,
가상 스레드를 다른 캐리어 스레드로 옮기면 락 소유자가 바뀌기 때문에 jvm에서 거부하는 것
이렇게 되면 캐리어 스레드를 반환하지 못해 ForkJoinPool의 이점이 사라진다.
그래서 ReentrantLock 라는 게 있고, 이건 가상 스레드 레벨에서 동작하여 unmount가 가능하다.

```java
ReentrantLock lock = new ReentrantLock();
  lock.lock();
  try {
      doBlockingIO(); // unmount 가능
  } finally {
      lock.unlock();
  }
```

그리고 java21에서는 이러한 문제가 이슈지만, java24에 가서는 이러한 문제가 해결되었다고 한다.(JEP 491)


# 2. 가상 스레드 생성

가상 스레드를 생성하는 방법은 크게 3가지다.

## 2-1. startVirtualThread()

가장 단순한 방법. Runnable을 인자로 받아 즉시 실행한다.

```java
Thread thread = Thread.startVirtualThread(() -> {
    System.out.println("Hello from virtual thread!");
});
thread.join();
```

> 주의: 가상 스레드는 기본적으로 데몬 스레드다.
> 메인 스레드가 종료되면 JVM이 함께 종료시키므로, join()으로 완료를 기다려야 한다.

## 2-2. Thread.ofVirtual() - Thread Builder API

생성과 실행을 더 세밀하게 제어할 수 있다.

```java
// 생성과 동시에 실행
var thread = Thread.ofVirtual().start(() -> System.out.println("Hello world!"));
thread.join();

// 생성과 실행 분리
var unstarted = Thread.ofVirtual().unstarted(() -> System.out.println("Hello world!"));
unstarted.start(); // 필요할 때 실행
```

## 2-3. Executor 사용

기존 코드베이스에서 가장 쉽게 전환할 수 있는 방법.
태스크 하나당 가상 스레드 하나를 생성한다.

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<String> future = executor.submit(this::callService);
    // future 처리
}
```

대규모 프로젝트에서 기존 ExecutorService 코드를 그대로 유지하면서 가상 스레드로 전환할 때 유용하다.

# 3. 가상 스레드 심화

## 3-1. 인터럽트 처리

가상 스레드는 Thread 클래스의 인스턴스이므로, 인터럽트 처리는 플랫폼 스레드와 동일하게 interrupt() 메서드로 처리.

### 인터럽트 처리 흐름(책 코드 예시)

① 빌더 패턴으로 플랫폼 스레드 또는 가상 스레드를 생성한다.<br>
② 스레드가 블로킹 연산 중 인터럽트되면 InterruptedException을 잡는다.<br>
③ 메인 스레드는 워커 스레드가 일정 횟수 반복을 마칠 때까지 대기한다.<br>
④ 워커 스레드를 인터럽트하여 예외 발생 및 뒤처리를 유발한다.<br>

### 플랫폼 스레드와의 차이점

인터럽트 API는 동일하지만, Thread.sleep() 같은 블로킹 연산 처리 과정에 결정적인 차이가 있음.

- 플랫폼 스레드: blocking 중에도 OS 스레드를 점유한 채 대기
- 가상 스레드: blocking을 만나면 제어권을 캐리어 스레드에게 반환 → 다른 가상 스레드 실행 가능(여기서 이점이 나온다!)

## 3-2. 스레드 그룹

모든 가상 스레드는 단 하나의 스레드 그룹에만 속함.
getThreadGroup()을 호출하면 항상 동일한 ThreadGroup 인스턴스가 반환되며, 다른 스레드 그룹을 지정하는 API는 없다.

가상 스레드는 다음과 같은 고정된 특징을 가짐.

- 우선순위: 항상 NORM_PRIORITY로 고정되며, setPriority()를 호출해도 변경되지 않는다.
- 데몬 상태: 항상 데몬 스레드이며, setDaemon()을 호출해도 변경되지 않는다.

## 3-3. Thread API 변경 사항

가상 스레드 도입으로 Thread API에도 생긴 변화

추가된 메서드:
- isVirtual(): 해당 스레드가 가상 스레드인지 판별한다.
- join(Duration), sleep(Duration): Duration을 인자로 받는 편의 메서드가 추가됐다.
- threadId(): 기존 getId()는 사용 중단됐고 final인 threadId()로 대체됐다.

사용 중단된 메서드:
- stop(), suspend(), resume(), countStackFrames()는 Java 20부터 호출 시 UnsupportedOperationException을 던진다. 원래부터 안전하지 않아 Java 1.2부터 사용 중단 예정이었다.

제약 사항:
- Thread.getAllStackTraces()는 플랫폼 스레드의 스택 트레이스만 반환하고 가상 스레드는 포함하지 않는다.
- 가상 스레드를 어떤 캐리어 스레드가 실행 중인지 확인하는 방법은 현재 없다.

# 4. 가상 스레드 생성 시연

## 4-1. 확장성 비교

책에선 10,000개의 태스크를 각각 1초씩 sleep하는 시나리오를 가정한다.

- newVirtualThreadPerTaskExecutor(): 최소한의 OS 스레드만으로 10,000개의 가상 스레드를 문제없이 처리
- newCachedThreadPool(): 10,000개의 OS 스레드를 생성하려다 비정상 종료
- newFixedThreadPool(200): 200개의 플랫폼 스레드로 처리하면 대부분의 태스크가 순차 처리되어 완료 시간이 급증

## 4-2. 처리량과 확장성

가상 스레드는 더 빠르게 실행되도록 설계된 것이 아니라 더 높은 확장성을 제공하도록 설계되었다.
리틀의 법칙(Little's Law)에 따라, 작업 속도가 아닌 더 높은 동시성 처리 능력으로 처리량을 높인다.

가상 스레드가 특히 효과적인 상황:
- 수천 개 이상의 동시 작업이 있는 경우 (대용량 웹 서버 등)
- CPU 집중 작업보다 I/O 대기 시간이 많은 작업 부하

반대로 CPU 집중적인 작업에서는 이점이 크지 않으며, 동시성 문제의 만병통치약은 아니다.

## 4-3. 리틀의 법칙(Little's Law)과 가상 스레드

### 리틀의 법칙이란?

큐잉(queueing) 시스템의 성능을 설명하는 근본 원칙으로, 지연 시간(latency), 동시성(concurrency), 처리량(throughput) 세 요소 사이의 수학적 관계를 정의.

```
처리량(λ) = 동시성(N) / 응답 시간(d)
```

- 처리량(λ): 단위 시간당 완료되는 요청의 평균 수
- 동시성(N): 동시에 처리 중인 항목의 평균 수
- 응답 시간(d): 단일 항목이 시작부터 완료까지 걸리는 평균 시간

이 법칙은 시스템의 성격에 구애받지 않는다.
동시성 단위가 스레드인지, CPU 코어인지 상관없이 동일하게 적용된다.

### 가상 스레드와의 연관

전통적인 스레딩 모델은 OS 수준의 스레드 수 제약으로 N(동시성)을 늘리기 어렵다.
N을 늘릴 수 없으면 처리량을 높이려면 d(응답 시간)를 줄이는 수밖에 없는데, I/O 집중 작업에서는 이것이 개발자의 통제 범위 밖인 경우가 많다.

가상 스레드는 프로그래밍 모델을 바꾸지 않고도 N을 크게 높일 수 있음.
즉, d를 줄이지 않고도 처리량(λ)을 높이는 것이 가상 스레드의 핵심 가치.

### 실질적인 의미

가상 스레드는

1. 대기 시간이 긴 I/O 집중적인 작업이 많은 경우, 해당 작업을 처리하는 코드 작성 방식을 바꾸지 않고도 처리량을 상당히 높일 수 있다.

2. 가상 스레드를 활용함으로써 결과적으로 리틀의 법칙에서 N(동시성)을 늘려서 궁극적으로 처리량을 높일 수 있다.

3. 하지만 가상 스레드가 모든 성능 문제의 만병통치약은 아니라는 점에 유의해야 한다.
I/O가 병목 현상의 원인일 때는 가상 스레드를 사용해서 처리량을 극적으로 개선할 수 있다.
그러나 CPU 집중적인 작업량(CPU intensive한 작업)이 병목 현상의 원인일 때는 가상 스레드가 전혀 도움이 되지 않는다.

정리하면 리틀의 법칙과 가상 스레드의 실질적인 의미를 이해하면, 언제 어떻게 가상 스레드를 사용해야 애플리케이션의 확장성과 성능을 높일 수 있는지 더 명확하게 알 수 있다.
